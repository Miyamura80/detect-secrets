## Codebase Patterns
- Use `source .venv/bin/activate && maturin develop` to build and install the Python package
- The Python crate is at `crates/python/` and depends on `crates/engine/` for pure Rust logic
- PyO3 module name must match `lib.name` in `crates/python/Cargo.toml` (currently `detect_secrets_rs`)
- maturin config lives in root `pyproject.toml` with `manifest-path` pointing to `crates/python/Cargo.toml`
- `cargo clippy` shows a benign `cdylib` warning on musl targets - this is expected and doesn't affect maturin builds
- Always run `cargo test -p engine -p detect-secrets-rs` to test both crates
- For negative lookaheads `(?![...])` (unsupported by Rust `regex`), use `(?:[^...]|$)` as a non-capturing boundary assertion — consumes the next char but doesn't affect capture groups
- All detector plugin structs must derive `Clone` for PyO3 wrapper compatibility (`Regex` implements `Clone`)
- Cloud/infra detectors pattern: struct with `Vec<Regex>`, `impl SecretDetector` + `RegexBasedDetector`, `impl Default`, `#[derive(Clone)]`
- The `detect-secrets/` submodule contains the original Python Yelp/detect-secrets for reference
- Git identity must be configured locally (`git config user.email/name`) - no global config exists
- PyO3 wrapper pattern: create a `crates/python/src/<module>.rs` file with a `#[pyclass]` wrapper around the engine struct, then register in `crates/python/src/lib.rs` via `m.add_class::<>()`
- For converting between Python dicts and serde_json Values, use helper functions `pydict_to_json_value` / `json_value_to_pydict` in the python crate
- The existing test in `potential_secret.rs` for `hash_secret("my_secret")` had wrong expected value - was fixed to `7585d1f7ceb90fd0b1ab42d0a6ca39fcf55065c7`
- For PyO3 modules with only functions (no classes), use a `pub fn register(m: &Bound<'_, PyModule>) -> PyResult<()>` pattern and call `entropy::register(m)?` in lib.rs
- Rust `regex` crate does NOT support lookbehinds (`(?<=...)`). Use `(?:^|\W)` instead of Python's `(?:(?<=\W)|(?<=^))`
- For regex patterns containing single quotes in raw strings, use `r#"..."#` syntax to avoid Rust lexer confusion
- The `regex_analyze_string` free function implements the default `RegexBasedDetector.analyze_string()` — plugins call it from their `SecretDetector::analyze_string()` impl
- Plugin trait pattern: `SecretDetector` (base) + `RegexBasedDetector` (regex-specific) traits in `crates/engine/src/plugin.rs`
- For `(?!\w)` negative lookahead (unsupported by Rust regex): use `(?:\W|$)` consuming boundary. For patterns WITHOUT capture groups, wrap the token in `(...)` so `regex_analyze_string` returns group 1 only
- JwtTokenDetector overrides `analyze_string()` to filter candidates — any plugin needing custom validation can do this instead of using `regex_analyze_string` directly
- Filters are organized as a `filters` submodule: `crates/engine/src/filters/{mod,heuristic,allowlist,common}.rs`
- Use `once_cell::sync::Lazy` for static regex and HashSet initialization in filter modules
- Allowlist filter uses pre-compiled regexes for 6 comment styles (Python, Go, C, VB.NET, SQL, XML) with both same-line and nextline pragma variants
- `is_line_allowlisted()` takes `previous_line` as a flat string parameter (Rust doesn't have Python's CodeSnippet context object yet)
- YAML files are optimized to only check `#` comment style; all other file types check all 6 comment styles
- Settings module (`crates/engine/src/settings.rs`) provides global singleton via `once_cell::sync::Lazy<Mutex<Settings>>` — use `get_settings()` / `get_settings_mut()` for thread-safe access
- Plugin registry: `builtin_plugins()` returns all 27 built-in plugins with class names and factory functions; `get_mapping_from_class_name()` provides `HashMap<&str, PluginFactory>` lookup
- `TransientSettingsGuard` is the Rust equivalent of Python's `transient_settings()` context manager — RAII guard restores settings on drop
- PyO3 `bool.into_pyobject(py)?` returns a `Borrowed` reference — must use `.to_owned()` before `.into_any().unbind()` to avoid move errors
- For nested JSON ↔ Python conversion, the `settings.rs` PyO3 module has its own `json_value_to_py()` / `py_to_json_value()` helpers that handle arrays, nested dicts, and all Value types
- SecretsCollection uses `HashMap<String, HashSet<PotentialSecret>>` — Python's `defaultdict(set)` maps to `HashMap::entry().or_default()`
- For mutating items in a HashSet (e.g., merge), extract into HashMap by identity key, mutate, then rebuild — HashSet doesn't allow mutable access to elements
- `secrets_collection.rs` PyO3 has its own `json_value_to_py()` / `pydict_to_json_value()` helpers that handle nested arrays and dicts recursively (needed for results format)
- Scan pipeline lives in `crates/engine/src/scan.rs` — `scan_file()`, `scan_line()`, `scan_diff()`, `get_files_to_scan()` are the public entry points
- For diff scanning, skip `IsInvalidFile` filter — files in a diff may not exist on the local filesystem
- `ScanFilterContext::from_settings()` captures all filter config (regexes, wordlist) at scan start — avoids repeated settings lookups during line iteration
- `is_regex_based()` heuristic: check `plugin.secret_type()` — High Entropy and Keyword plugins are NOT regex-based (needed for `is_likely_id_string` filter)
- `CodeSnippet` struct provides previous-line context for allowlist detection (mirrors Python's `CodeSnippet` from `util/code_snippet.py`)
- `BASELINE_VERSION` constant (`1.5.0`) in `baseline.rs` — used for baseline format compatibility with Python detect-secrets (separate from crate `VERSION` which is `0.1.0`)
- Enable `serde_json` `preserve_order` feature in engine crate — JSON key insertion order is preserved (needed for baseline output matching Python format)
- `get_files_to_scan()` returns paths relative to root — `create()` must resolve back to absolute paths for `scan_file()` to read from disk
- Tests using `default_settings()` must keep the guard alive: `let _guard = settings::default_settings()` (not `let _ = ...` which drops immediately)
- Tests using `get_files_to_scan()` with empty paths scan the root directory — never use `/tmp` as root (recursively walks all files), use a dedicated empty temp directory
- For parallel scanning, use `ParallelScanContext` pattern: capture settings into `Arc` wrappers once before `rayon::par_iter()` — avoids global mutex contention
- PyO3 bindings for CPU-bound parallel code must use `py.allow_threads(|| ...)` to release the GIL
- Mixed Python/Rust project: set `python-source = "python"` in `[tool.maturin]` and create `python/<package>/` with `__init__.py`
- For CLI entry points, add `[project.scripts]` to pyproject.toml pointing to Python modules that import the Rust extension
- Global settings mutation from Python CLI: use `global_*` PyO3 functions (e.g. `global_disable_plugins`, `global_set_filter`) that modify the singleton directly
- PySecretsCollection's `__iter__` returns a PyList (not a Python iterator) — iterate with `secrets.files` + `secrets[filename]` pattern instead
